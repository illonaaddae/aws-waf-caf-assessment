# WAF Assessment — Five-Pillar Evaluation

## Task 2: AWS Well-Architected Framework Assessment Table

**Workload:** Two-Tier Web Application (Apache + Node.js / MySQL)
**Migration Target:** Amazon Web Services
**Assessment Date:** March 1, 2026

## Evaluation Criteria Applied

For each pillar, the assessment identifies:

- **One strength** in the current architecture that carries forward to AWS
- **One area for improvement** with specific reasoning grounded in the workload's observed weaknesses
- **An actionable improvement recommendation** — specific, not categorical
- **The supporting AWS service(s)** correctly matched to the pillar

| **Operational Excellence** | **Strength:** The two-tier architecture is well-understood by the operations team, keeping the scope of migration cognitively manageable and reducing the risk of misunderstood dependencies during cutover planning. **Weakness:** All deployments are executed manually via SSH with no audit trail, version control, or rollback capability. Operational failures are discovered reactively — there are no runbooks, no centralised logs, and no defined incident response procedures.                                                                                                                                                                                                                                                                                                                                                                     | Adopt infrastructure-as-code (IaC) so that every environment change is versioned, peer-reviewed, and replayable. Implement a CI/CD pipeline to eliminate manual deployment risk and introduce rollback by default. Centralise application and infrastructure logs to a managed log service. Define operational runbooks for the most likely failure scenarios (e.g., failed deployment, database connection loss, certificate expiry).                                                                                                                                                                      | AWS CodePipeline + CodeDeploy (automated release pipeline); AWS CloudFormation / AWS CDK (IaC); Amazon CloudWatch Logs (centralised log aggregation); AWS Systems Manager OpsCenter (runbook management and operational automation)                                                                         |
| **Security**               | **Strength:** The separation of the application and database into distinct server tiers provides a structural foundation for network segmentation — this boundary translates directly into a VPC subnet model on AWS where the database tier can be placed in an isolated subnet with no inbound internet route. **Weakness:** Security groups are overly permissive (SSH port 22 open to broad source ranges). No encryption exists at rest or in transit. Root or shared OS credentials are in use across both servers, violating least-privilege principles. Database credentials are stored in application configuration files with no secrets management.                                                                                                                                                                                                | Enforce least-privilege security groups restricting inter-tier traffic to only the required ports and sources. Enable TLS for all data in transit via managed certificates. Replace all static credentials with IAM roles applied to compute resources. Encrypt all persistent data at rest. Centralise secrets management with automated credential rotation to eliminate stored plaintext credentials. Deploy a web application firewall at the network edge to filter malicious HTTP/S traffic before it reaches the application tier.                                                                   | AWS IAM (roles and policies — no static keys on servers); AWS Certificate Manager (managed TLS); AWS Secrets Manager (credential storage and rotation); AWS KMS (encryption at rest for RDS, EBS, S3); AWS WAF (OWASP Top 10 managed rule groups)                                                           |
| **Reliability**            | **Strength:** The application frontend is stateless — session state is not persisted locally on the web server. This means instances can be replaced or scaled horizontally without application logic changes, making multi-AZ distribution achievable with minimal refactoring effort. **Weakness:** The entire workload runs on two physical servers in a single location with no redundancy at any tier. Any hardware failure, OS-level fault, or AZ-equivalent disruption causes a complete and indefinite outage. No automated failover exists for the database. No backup strategy is in place, meaning the RPO is effectively infinite and the RTO is undefined.                                                                                                                                                                                       | Deploy the workload across a minimum of two AWS Availability Zones. Place the application tier behind an Application Load Balancer connected to an Auto Scaling Group spanning both AZs. Migrate the database to Amazon RDS in Multi-AZ configuration to enable synchronous replication and automated failover without manual intervention. Define explicit RTO and RPO targets and implement automated backup policies with scheduled, documented restoration tests.                                                                                                                                       | Elastic Load Balancing / ALB (cross-AZ traffic distribution and health-check-based routing); Amazon RDS Multi-AZ (synchronous standby with automated promotion); AWS Backup (centralised backup policies with defined retention); Amazon Route 53 (health-check-based DNS failover for regional resilience) |
| **Performance Efficiency** | **Strength:** The decoupled two-tier model means the application and database tiers can be independently scaled on AWS. The web tier can be horizontally scaled without any schema or database changes, and the database tier can be vertically or horizontally scaled independently of the application code. **Weakness:** No autoscaling is configured — traffic spikes are accommodated only by persistent over-provisioning of static capacity, which also incurs ongoing cost waste. The database is a single vertically-constrained server with no read offload. Static assets (images, CSS, JS bundles) are served directly from the application server, consuming compute resources and increasing origin response latency for geographically distributed users.                                                                                      | Implement an EC2 Auto Scaling Group for the application tier with a dynamic scaling policy aligned to ALB request metrics, ensuring compute capacity tracks actual demand. Offload all static asset delivery to a content delivery network to eliminate unnecessary application server load and reduce latency at the edge. Evaluate read-heavy database query patterns post-migration for ElastiCache adoption. After 30 days of production data, run AWS Compute Optimizer to rightsize all EC2 and RDS instances based on observed utilisation.                                                          | Amazon EC2 Auto Scaling (demand-driven horizontal scaling); Amazon CloudFront (CDN for static asset delivery); Amazon ElastiCache for Redis/Memcached (database query caching, Phase 2); AWS Compute Optimizer (rightsizing recommendations based on CloudWatch metrics)                                    |
| **Cost Optimization**      | **Strength:** Migration to AWS converts fixed capital expenditure (server hardware, data centre space, power, maintenance contracts) into variable operating expenditure billed per unit of consumption. With autoscaling in place, the workload pays only for the compute actively serving traffic — eliminating the structural idle spend inherent in always-on on-premises hardware. **Weakness:** Without a cost governance framework established from day one, the migration risks replicating on-premises over-provisioning in the cloud. Absent tagging, teams lose cost attribution by environment and team. Absent budgets and alerting, unexpected spend goes undetected. The default behaviour — provisioning without optimisation — produces a "lift and shift cost trap" where cloud bills exceed on-premises costs with no visibility into why. | Enforce a mandatory resource tagging policy from the first provisioned resource, covering at minimum: `environment`, `owner`, `cost-centre`, and `project`. Enable AWS Cost Explorer and configure AWS Budgets with threshold alerts before any significant workload is deployed. After 30–60 days of stable production usage, evaluate EC2 Savings Plans or Reserved Instances for baseline compute. Schedule monthly FinOps review sessions using Cost Explorer and Trusted Advisor findings. Implement S3 lifecycle policies to transition infrequently accessed log data to lower-cost storage classes. | AWS Cost Explorer (spend analysis and forecasting); AWS Budgets (threshold and forecast alerting); AWS Trusted Advisor (cost optimisation checks); EC2 Savings Plans / Reserved Instances (commitment-based discounting for baseline workloads); AWS Cost Anomaly Detection (ML-based anomaly alerting)     |
